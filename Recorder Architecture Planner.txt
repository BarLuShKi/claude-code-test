# Section 1 — Product Architecture: Adversarial Review & Hardening

> **Scope (per user redirect):** Focus on **Section 1 (Product Architecture)** only. Adversarially review it in isolation from many angles, harden it, then organize the result and commit it to git.

---

## Context

The original Section 1 sketched a single-purpose, "press record and walk away" multitrack recorder with four pillars (Trust, Speed, Awareness, Recoverability), three personas, a four-mode flow, and a short non-goals list. Treated in isolation and attacked from multiple directions, it holds up directionally but has **material gaps** that would bite a real broadcast/reality-TV deployment. This document records the attack, then a hardened Section 1, then how it lands in git.

---

## Part A — Adversarial Review (by angle)

Each finding: **severity** (Critical / High / Medium / Low) + the fix folded into Part B.

### A1 — Competitive positioning (High)
Original states "inspired by Boom Recorder" but never says *why someone picks this over the alternatives*: Boom Recorder, Reaper-as-recorder, Metacorder, Sound Devices Wave Agent (metadata), or a hardware recorder (Sound Devices/Cantar/Aaton/Nomad). Without an explicit wedge, the product is "another recorder."
**Fix:** add a positioning statement — software-defined, high-density (Dante-class) iso recorder with first-class metadata/naming/post hand-off and operational awareness, on commodity Apple Silicon. Differentiator = reliability + workflow + scale at software cost, not yet-another-feature-set.

### A2 — The headline promise overclaims (Critical)
"Press record, walk away 12 hours, get perfect files" is **unfalsifiable and unsafe** as stated. A single Mac mini faces power loss, kernel panics, thermal throttle, OS auto-updates, sleep/App Nap, Dante network blips, disk-full. The promise is only credible when paired with explicit *system-level* dependencies and *measurable guarantees*.
**Fix:** (1) replace the vague promise with **explicit product guarantees/SLOs** (below); (2) add a **System Reliability Dependencies** subsection (appliance posture, sleep/App Nap inhibition, OS-update/Time-Machine policy, UPS expectation, NTP policy).

### A3 — Missing reliability dependencies the app itself must own (Critical)
The app must actively defend its own runtime: prevent **App Nap** and **system/display sleep** while armed/recording, refuse cloud/network/Time-Machine destinations (already noted for §7 but it's a *product* principle), and warn on low power / thermal pressure. None of this was in Section 1.
**Fix:** new subsection "System Reliability Dependencies" + elevate "Capture is sacred" to a first-class principle.

### A4 — Take-based vs. continuous paradigm tension (High)
Section 1 implicitly leans **film/scripted** (take/roll). Reality TV, live events, and broadcast are predominantly **continuous iso recording** with marker-based logging and rollover, not discrete takes. The product targets *both*, but Section 1 never names the tension — which would mis-shape the whole UX and data model.
**Fix:** state both operating paradigms explicitly (Take mode vs Continuous mode) as a product-level concept.

### A5 — Pre-record buffer is absent (High)
Every serious production recorder (Boom Recorder, Sound Devices) offers a **pre-record / pre-roll buffer** (capture N seconds *before* the operator hits record). Omitting it from the product mode model is a competitive and operational miss.
**Fix:** add a **Standby (pre-record buffering)** state to the mode model.

### A6 — No degraded/error state in the mode model (High)
The "if a subsystem fails, recording continues and is logged" philosophy has **no state to represent it**. Setup→Armed→Recording→Review can't express "Recording (Degraded): mirror lost / LTC lost / disk warning."
**Fix:** add **Recording (Degraded)** as an explicit operating state and define the failure hierarchy.

### A7 — Redundancy / single-point-of-failure (High)
"Walk away 12 hours" on one box ignores that broadcast/reality runs redundant recorders (A/B) and UPS. Section 1 doesn't state whether multi-machine redundancy is in product scope (even future).
**Fix:** add a redundancy stance — mirrored disks (near-term, already in spec) + a documented "run a second instance as backup recorder" posture, with full A/B chase deferred but acknowledged.

### A8 — Personas are incomplete and conflated (Medium)
"Sound mixer / A2" conflates roles; reality/live use a **central audio control room operator** and **sound utility**, plus the **rental/facility tech** who builds presets, the **producer/showrunner** who only wants the dashboard, and the **Dante/IT network admin**. Also: buyer (production company) ≠ user.
**Fix:** expand and de-conflate personas; separate buyer from users.

### A9 — Missing pillar: interoperability / post hand-off (Medium)
BWF/iXML/markers/CSV/daily-folders are a *core value* (post integration) but aren't reflected in the pillars. The pillar set under-weights "fits the production→post pipeline."
**Fix:** add a fifth pillar — **Interoperability / clean hand-off to post**.

### A10 — Confidence monitoring output is unaddressed (Medium, open decision)
"Input metering only" + spot-check playback omits **live confidence monitoring** (headphone/PFL/solo) that production-sound operators expect. This is a genuine product-scope decision, not an oversight to silently keep out.
**Fix:** surface as an explicit **open product decision** (in-scope live monitoring vs. metering-only), with a recommendation.

### A11 — "No mixer" vs. per-track gain inconsistency (Low)
Non-goals say "no mixer," but the product has per-track gain. Ambiguous.
**Fix:** clarify gain is **record trim**, not a monitor mix bus.

### A12 — System clock / time integrity (Medium)
Wall clock drives naming, daily folders, and midnight logic; NTP jumps / DST / timezone changes can corrupt names and confuse midnight detection (which §8 separates from LTC corruption).
**Fix:** add a product-level time-integrity stance (recommend stable/NTP-disciplined-but-no-step clock during record; treat wall-clock and media-clock as distinct).

### A13 — Appliance vs. general-purpose app (Medium)
Reliability is far higher if the target Mac mini runs as a **dedicated appliance** (auto-launch, auto-recover, minimal other software, kiosk-ish) rather than a general desktop app among many. This is a product-architecture choice that was never made.
**Fix:** state the **appliance-first deployment posture** as the recommended operating model.

### A14 — Accessibility / colorblind-safe metering / localization (Low)
"Modern UX" for international crews implies colorblind-safe meters, large-text legibility at a distance, and localization. Not mentioned.
**Fix:** add a short UX-principles note at product level.

### A15 — Guarantees are not measurable (Critical, ties A2)
"Trust" and "perfect files" can't gate development. Product architecture should declare testable guarantees that become acceptance gates.
**Fix:** **Product Guarantees (SLOs)** subsection with falsifiable statements.

## Part B — Hardened Section 1 (Product Architecture)

### 1.0 Positioning
A **software-defined, high-density isolated-track recorder** for Apple Silicon: Dante-class channel counts, first-class metadata/naming/post hand-off, and broadcast-grade operational awareness — at commodity-hardware cost. It competes with Boom Recorder/Metacorder/Reaper-as-recorder and offloads from hardware recorders where channel density, metadata flexibility, and dashboarding matter more than a self-contained device. **Wedge = reliability + workflow + scale, not feature count.**

### 1.1 What it is
A single-purpose, always-trustworthy recorder whose **prime directive is "Capture is sacred"**: the capture path must never drop a sample; *every other subsystem (UI, mirror, LTC, network, dashboard) is best-effort and may degrade, but never stops or corrupts capture.*

### 1.2 Product Guarantees (SLOs) — testable, become acceptance gates
- **G1 No-loss:** zero dropped samples under the supported load envelope (channel count × SR × bit depth on the certified device matrix).
- **G2 Crash-valid:** every record press yields a recoverable, openable file even on power loss / kill-9 mid-write.
- **G3 TC-accurate:** per-file start timecode and BWF `TimeReference` are sample-accurate when LTC is present.
- **G4 Degrade-don't-die:** any single non-capture subsystem failure leaves capture running and is logged + surfaced.
- **G5 Long-run-stable:** bounded memory/CPU/thermals across multi-hour and multi-day soak (no leaks, no drift-induced xruns).

### 1.3 Product Pillars
1. **Trust** — capture never glitches; guarantees above are gated by tests.
2. **Continuity / uptime** — built for multi-hour/multi-day long-form.
3. **Speed of setup** — presets, track templates, token naming, auto take/roll.
4. **Operational awareness (not alarm fatigue)** — leveled INFO/WARNING/CRITICAL with hysteresis + filtering.
5. **Recoverability** — any interruption yields valid, recoverable files.
6. **Interoperability** — clean hand-off to post (BWF/iXML, markers, CSV, daily folders).

### 1.4 Operating Paradigms (both supported)
- **Take mode** (film/scripted): discrete takes, roll/take/scene, slate-style metadata.
- **Continuous mode** (reality/live/broadcast): long continuous iso record, marker-based logging, time/size/TC rollover.
The UX and data model must serve both; the difference is primarily in counters, naming defaults, and the logging emphasis.

### 1.5 Operating Modes (state machine)
`Setup → Armed → Standby(pre-record buffering) → Recording → Recording(Degraded) ↔ Recording → Stopped → Review`
- **Standby/Pre-record:** rolling N-second pre-buffer so record captures audio *before* the press.
- **Recording(Degraded):** capture continues; a non-capture subsystem (mirror/LTC/disk-warning/device-loss) is impaired and surfaced.
- **Review:** spot-check playback, **mutually exclusive with Recording** (isolated engine).
- **Overlays:** **Lock Mode** (block config/routing/device/metadata edits) and **Record Safe Mode** (guard Stop/Quit/Close/Change-project/Change-destination), enforced in the command layer across *all* control surfaces.

### 1.6 Failure Hierarchy (defines G4)
- **Tier 0 — Sacred (must never stop/corrupt):** input capture, FIFO drain, primary-disk write, crash-valid headers.
- **Tier 1 — Important (degrade, keep recording):** LTC, mirror disk, metering, health monitor, event log.
- **Tier 2 — Peripheral (may pause/fail freely):** remote dashboard, exports, UI cosmetics, control-surface links.

### 1.7 System Reliability Dependencies (the app owns these)
- Inhibit **App Nap** and **system/display sleep** while Armed/Recording.
- **Destination policy:** local/external physical volumes only; refuse cloud-synced (OneDrive/iCloud/Dropbox), network, and Time-Machine-target volumes.
- **Power:** detect on-battery / low-power / thermal pressure and warn; recommend UPS for the appliance.
- **Time integrity:** treat wall-clock and media-clock as distinct; recommend a disciplined-but-non-stepping system clock during record; handle DST/timezone without breaking naming or midnight logic.
- **OS hygiene (deployment guidance):** disable auto OS updates and Time Machine on the record machine during productions.

### 1.8 Deployment Posture
**Appliance-first:** recommend running the target Mac mini M4 as a dedicated recorder appliance (auto-launch, auto-recover after restart, minimal other software, kiosk-like). It also runs as an ordinary app, but appliance mode is the reliability-optimal default.

### 1.9 Redundancy Stance
- **Near-term (in spec):** mirrored recording to a secondary disk (best-effort; primary never blocked).
- **Operational:** documented "run a second instance/machine as a backup recorder" posture.
- **Deferred:** full A/B chase-sync between instances — acknowledged, not in early scope.

### 1.10 Personas (de-conflated)
- **Buyer:** production company / facility (in-house first).
- **Production sound mixer / recordist** (Take mode, film/scripted).
- **Audio control-room operator + sound utility** (Continuous mode, reality/live).
- **Broadcast/OB engineer** (high-count Dante, LTC, long runs, disk planning).
- **Rental/facility tech** (builds device/track/session presets).
- **Producer / showrunner** (read-only remote dashboard consumer).
- **Dante / IT network admin** (network + clocking).
- **Post / DIT** (BWF/iXML, daily folders, markers, take logs).

### 1.11 UX Principles (product-level)
Modern macOS Dark/Light; **colorblind-safe, high-legibility metering** readable at a distance; consistent command model across keyboard/Stream Deck/OSC/HTTP; localization-ready.

### 1.12 Scope & Non-Goals
- **In scope:** capture, per-track **record-trim gain** + delay + bypass, input metering, markers, isolated spot-check playback, post-record exports, metadata/naming/folders, LTC awareness, health/recovery, mirrored record (near-term), remote dashboard (future).
- **Non-goals:** not a DAW; no editing; no plugins/effects beyond delay+gain; **per-track "gain" is record trim, not a monitor mix bus**; not a routing matrix/mixer.
- **Open product decision (A10):** *live confidence monitoring* (headphone/PFL/solo output) — currently metering-only. **Recommendation:** treat basic confidence monitoring (selectable solo/headphone bus) as in-scope for production-sound credibility, scheduled after MVP; confirm before locking the UX.

# Section 2 — Software Architecture: Adversarial Review & Hardening

> **Scope:** Focus on **Section 2 (Software Architecture)** only. Attack it in isolation from many angles, then harden it.

## Context (what Section 2 originally said)
A layered design — Presentation → Application/Command → Engine façade (publishes snapshots) → four engines (Capture RT, Recording, Disk Writer pool, LTC) → mockable Platform abstraction — with four cross-cutting rules: RT/non-RT firewall, one-way snapshot publication, single command choke point, everything mockable. Directionally sound, but several load-bearing details are unspecified or quietly wrong under attack.

## Part A — Adversarial Review (by angle)

### A1 — Threading model is named but not specified (Critical)
The diagram lists threads but defines no **thread inventory, priority policy, or non-blocking contract**. Unspecified: who owns each thread, real-time vs high vs normal priority, the rule that the RT thread *never* waits on any other thread, priority-inversion avoidance, and start/stop **shutdown ordering** (a classic use-after-free source).
**Fix:** add an explicit Thread Inventory + priority/ownership/lifecycle table and the "RT waits on no one" invariant.

### A2 — Two distinct planes are conflated: control vs data (Critical)
"Engine publishes snapshots at ~30 Hz" lumps slow status with the **hot meter/waveform data path**. At 128 tracks the meter plane wants direct lock-free per-track atomics, not a fat status struct copy.
**Fix:** split a **Control/Status plane** (commands + status snapshots) from a **Data plane** (audio FIFOs + per-track meter atomics). Different mechanisms, different rates.

### A3 — Snapshot mechanism + memory reclamation underspecified (High)
"Double-buffered / atomic seqlock" hand-waves the hard part: seqlock readers spin; pointer-swap double-buffering needs safe reclamation (ABA / who frees the old snapshot). 
**Fix:** pin a concrete, RT-safe scheme — **triple-buffer with atomic publish index** (no reader spin, no allocation, no free-after-read race) for status; raw atomics for meters.

### A4 — How a command reaches the RT thread is missing (High)
"Commands go through the dispatcher" stops at the firewall. Live changes (arm, set gain/delay, bypass) must cross into the RT thread **without locks**.
**Fix:** define a **lock-free command FIFO** consumed by the engine at buffer boundaries, plus per-parameter atomics for cheap continuous values (gain/bypass). No control path takes a lock the RT thread could contend.

### A5 — No event/error bus; error model undefined (High)
EventLog and HealthMonitor exist as modules, but Section 2 shows **no channel** by which a writer-thread disk-full or an LTC dropout *reaches* them, and no error-handling strategy (exceptions are forbidden in RT). Status snapshots are one-way and continuous; discrete **events** (errors, markers, LTC events, rollover) need their own path.
**Fix:** add an explicit **lock-free event bus** (engine → app) for discrete events, and a stated error model: `Result`/error-code returns at boundaries, **no exceptions across the firewall or in RT**.

### A6 — No TimeAuthority / Transport component (High)
Sample-clock ↔ wall-clock ↔ timecode mapping is the backbone of BWF `TimeReference`, TC-rollover, markers, and midnight logic — yet "Clock" appears only as the *system* clock in the platform layer. There's no owner of the **sample-count master + sample↔TC↔wall map**.
**Fix:** introduce a **Transport / TimeAuthority** component (engine-level) owning the monotonic sample master and the time-domain mapping; LTC/Recording/Marker subsystems consume it.

### A7 — Layering inversion: engine writes the RecoveryJournal (Medium)
RecoveryJournal sits in the Application layer, but the **writer (engine layer) must update it during recording** — an engine→application dependency that violates the layering.
**Fix:** define journal/event sinks as **interfaces injected downward** (dependency inversion); the engine depends on `IJournalSink`/`IEventSink`, not on app-layer classes.

### A8 — Process model: a Tier-2 crash kills Tier-0 (High)
Everything is one process, so a fault in dashboard/export/UI can take down **capture** — directly contradicting "Capture is sacred."
**Fix:** state a process-isolation policy — run **exports out-of-process** (ffmpeg child) now, and put the **remote dashboard** (and ideally control-API ingress) behind an **XPC/separate process** so Tier-2 cannot crash Tier-0. Keep MVP simple but reserve the seam.

### A9 — FIFO overflow / backpressure policy undefined (High)
"Large buffers absorb disk stalls" is not a policy. What happens when a FIFO actually fills (disk can't keep up)?
**Fix:** define the **overflow contract**: RT never blocks; on full FIFO it marks an **xrun event** (counted, surfaced, logged) and follows a stated drop policy; sizing = "seconds of headroom" with the overflow path exercised in tests, not assumed impossible.

### A10 — Over-abstraction risk for a solo dev (Medium)
Façade + dispatcher + a dozen services is gold-plate-able. 
**Fix:** mark the **MVP-essential abstractions** (RT firewall, platform mocks, lock-free FIFO, snapshot/event bus, command FIFO) vs. **start-concrete-extract-later** (full service layer, façade richness). Build simple first; the firewall is the only abstraction that's non-negotiable from day one.

### A11 — Single source of truth / change-notification absent (Medium)
Session/Metadata/Naming/Counters all hold state; a metadata change must update the naming preview, etc. No defined ownership or observer model → drift/inconsistency.
**Fix:** name a **SessionModel** as single source of truth on the message thread with a change-notification (observer) mechanism feeding dependent services and the UI.

### A12 — Schema versioning / migration not architectural (Medium)
JSON sessions/presets and the SQLite journal need **version fields + migration** from v1, or early files break later.
**Fix:** add schema-version + forward-migration as a platform/storage concern.

### A13 — Control-API trust boundary missing (Medium)
Dashboard is read-only (good), but **OSC/HTTP control is read-write into Tier-0 commands**. No auth/bind-scope/rate-limit/authz stated.
**Fix:** define a **network trust boundary**: control ingress is authenticated, LAN-bound, rate-limited, routed through the *same* command choke point (so Record-Safe/Lock apply), and separated from the read-only dashboard surface.

### A14 — Startup / recovery / device-acquire sequencing undefined (Medium)
No boot orchestration: detect prior crash → recovery decision → acquire device → restore session → (never auto-arm before recovery is resolved).
**Fix:** add a **lifecycle/orchestration** description with explicit ordering and "no auto-arm before recovery decision."

### A15 — Engineering observability seam not called out (Medium)
RT-callback duration histograms, FIFO high-watermarks, writer latency feed both tests and HealthMonitor, but aren't an architectural cross-cutting concern.
**Fix:** add a **telemetry seam** (lock-free counters/histograms) as the shared source for internal metrics, tests, and user-facing health.

### A16 — Stack alignment not stated (Low)
The threading/FIFO mechanisms should map to the chosen stack (JUCE: `AbstractFifo`, `AudioIODeviceCallback`, message thread) and note deviations (custom writer threads vs `ThreadPool`).
**Fix:** one note aligning the model to JUCE primitives and where we go custom.

## Part B — Hardened Section 2 (Software Architecture)

### 2.1 Two planes
- **Data plane (hot):** audio input → per-track lock-free SPSC FIFOs → writer pool; per-track **meter atomics** read directly by UI/dashboard. No locks, no allocation, no syscalls.
- **Control/Status plane:** commands in (lock-free command FIFO + parameter atomics) and status out (triple-buffered status snapshot + lock-free **event bus** for discrete events).

### 2.2 Layers (with corrected dependency direction)
Presentation (GUI + read-only dashboard) → Application (SessionModel = single source of truth, command dispatcher with Record-Safe/Lock gating, services) → **Engine façade** → Engines (Capture RT, **Transport/TimeAuthority**, Recording, Disk Writer pool, LTC) → Platform abstraction (AudioDevice, SystemClock, FileSystem, DiskSpace, Storage[SQLite/JSON, versioned], Net). **Engines depend on injected interfaces (`IEventSink`, `IJournalSink`) — never on app-layer classes.**

### 2.3 Thread inventory & invariants
| Thread | Priority | Owns | Blocking allowed? |
|---|---|---|---|
| Audio callback | Real-time (CoreAudio) | capture, delay/gain, meter+clip taps, FIFO push, consume command FIFO at buffer edge | **Never** waits on any other thread |
| Writer pool (2–4) | High (non-RT) | FIFO drain, format, write, header patch, journal sink | May do I/O; never blocks RT |
| LTC decode | High | LTC tap → decode → events | No |
| Monitor/health | Normal | telemetry, leveling, snapshots | No |
| Message/UI | Normal | SessionModel, dispatch, rendering | Yes (off RT) |
| Net/dashboard | Low | read-only serve; control ingress via dispatcher | Yes (isolated) |

**Invariants:** RT waits on no one; no exceptions in RT or across the firewall; no allocation after Arm; defined thread **start/stop ordering** (acquire device last on start, release first on stop) to prevent use-after-free.

### 2.4 Cross-cutting concerns (added)
- **Event/error model:** discrete events via lock-free event bus; `Result`/error-codes at boundaries; errors classified by **failure tier** (Sacred/Important/Peripheral from §1.6).
- **Backpressure:** FIFO-full ⇒ RT marks counted xrun event + stated drop policy; never blocks.
- **TimeAuthority:** monotonic sample master + sample↔TC↔wall map; sole source for `TimeReference`, rollover, markers, midnight.
- **Telemetry seam:** lock-free counters/histograms feeding tests + health.
- **Persistence:** versioned schemas + migration; SessionModel as single source of truth with change-notification.
- **Network trust boundary:** authenticated, LAN-bound, rate-limited control ingress through the one command choke point; read-only dashboard separated.
- **Process isolation:** exports out-of-process now; dashboard/control ingress behind XPC/separate process so Tier-2 can't crash Tier-0.
- **Lifecycle:** boot = detect crash → recovery decision → acquire device → restore session; **no auto-arm before recovery resolved.**

### 2.5 Solo-dev pragmatism
Non-negotiable from day one: **RT firewall, lock-free FIFO, command FIFO + status/event channels, platform mocks, telemetry seam.** Start concrete and extract later: façade richness, full service layer, XPC isolation (reserve the seam, implement when dashboard/exports land). Don't gold-plate the layering before the capture tracer-bullet proves out.

## Verification (Section 2)
- Walkthrough: trace a "disk full at 128ch while recording with LTC" event end-to-end — RT marks xrun → event bus → health leveling → UI/dashboard + EventLog + journal — with **capture never stalling**. The hardened model must express every hop without a lock on the RT thread.

# Section 3 — Module Breakdown: Adversarial Review & Hardening

> **Scope:** Focus on **Section 3 (Module Breakdown)** only. Attack the module list in isolation, then re-cut it.

## Context (what Section 3 originally said)
A flat 19-row table (AudioDeviceManager, CaptureEngine, RecordingEngine, DiskWriterPool, LTCEngine, MeteringService, MetadataService, NamingEngine, CounterService, SessionStore, RecoveryJournal, HealthMonitor, EventLog, MarkerService, ControlSurfaces, RemoteDashboard, PlaybackEngine, ExportService, GUI/ViewModels) with a "responsibility" and a loose "thread context" column. Useful inventory, but it predates the §2 hardening, mixes altitudes, and hides what's reliability-critical.

## Part A — Adversarial Review (by angle)

### A1 — Modules implied by hardened §1/§2 are missing (Critical)
The breakdown doesn't include components the design now requires: **Transport/TimeAuthority**, **EventBus**, **CommandService** (the one choke point with Record-Safe/Lock gating — currently buried inside "ControlSurfaces"), **SessionModel** (in-memory single source of truth — distinct from the persistence-only "SessionStore"), **TelemetryService**, **PreRecordBuffer** (standby/pre-roll), **Destination/VolumePolicy** (local-only, cloud/network refusal, disk planning), **MirrorWriter**, **PresetService/TemplateService** (lumped into SessionStore), and an **AppCoordinator** (composition root / lifecycle). Optional **MonitoringOutput** if confidence monitoring is approved (§1 open decision).
**Fix:** add these as first-class modules.

### A2 — Altitude/granularity is inconsistent; criticality invisible (High)
A tiny RT-critical `CaptureEngine` sits in the same flat list as the huge `GUI/ViewModels`, with no grouping by **failure tier** (§1.6) or **plane** (§2.1). You can't see at a glance what must never fail.
**Fix:** regroup by **Tier (0/1/2)**, and tag **plane** + **RT-safety class** per module.

### A3 — Overlapping responsibilities / fuzzy ownership (High)
- **Crash-valid headers:** DiskWriterPool (header patching) vs RecoveryJournal — who owns recoverability? 
- **Rollover:** RecordingEngine "orchestration" vs DiskWriterPool "gapless rollover" — decide vs execute unclear.
- **Metering/clip:** the RT *tap* lives in CaptureEngine, aggregation in MeteringService, leveling in HealthMonitor — three owners, fuzzy seam.
- **Filename resolution:** NamingEngine + CounterService + MetadataService all feed it — which one *owns* "produce the final path"?
**Fix:** state explicit **decide-vs-execute / tap-vs-aggregate-vs-level** boundaries per pair.

### A4 — No dependency/interaction map (High)
A table can't be validated against §2's "no engine→app dependency" and Tier rules without showing **who talks to whom**.
**Fix:** add dependency rules + a short interaction list; assert **no Tier-0 module depends on a Tier-2 module**.

### A5 — Thread-context column is vague and omits RT-safety class (Medium)
"Any → serialized", "Message + control" don't say whether a module may be touched from the RT thread. (e.g., EventLog must be **never called from RT** — RT only posts to EventBus.)
**Fix:** replace with **thread owner + RT-safety class** (RT-safe / non-RT / never-from-RT).

### A6 — Test strategy per module not indicated (Medium)
Pure/deterministic modules (NamingEngine, CounterService, TimeAuthority math) deserve exhaustive unit tests; hardware-bound ones (AudioDeviceManager) need mocks + soak. The table doesn't distinguish.
**Fix:** tag each module's **primary test strategy**.

### A7 — No lifecycle owner / composition root (Medium)
§2 added boot/recovery sequencing and DI, but no module owns it.
**Fix:** add **AppCoordinator** (composition root: builds graph, injects interfaces, owns start/stop ordering, drives the boot→recovery→acquire→restore sequence).

### A8 — Process-model decisions not reflected (Medium)
ExportService is shown in-process ("background worker"); §2 hardened it to **out-of-process ffmpeg**. RemoteDashboard likewise XPC-isolated.
**Fix:** update both to reflect supervision of isolated child processes.

### A9 — ControlSurfaces conflates trust levels (Medium)
Keyboard + Stream Deck + OSC + HTTP in one module spans local input and **network ingress** with very different trust. The §2 network trust boundary has no owner.
**Fix:** split **LocalControl** (keyboard / local Stream Deck) from **NetworkControlGateway** (OSC/HTTP: auth, bind-scope, rate-limit); both funnel into **CommandService**.

### A10 — Too many modules for a solo dev to scaffold up front (Medium, YAGNI)
~27 modules after additions; MVP (§18) needs a fraction.
**Fix:** tag each module with **phase** (MVP / TC / Workflow / Hardening / Later) per §17 so v1 stays small.

### A11 — PlaybackEngine device contention undefined (Low/Med)

# Section 4 — Risk Analysis: Adversarial Review & Hardening

## Context
Original had a 12-row risk table (R1–R12) plus a throughput sanity note. Good start, but it conflates probability with impact, omits whole risk classes, and has no scoring or ownership.

## Part A — Adversarial Review
- **A1 (High) No likelihood × impact scoring or residual risk.** A flat table can't prioritize. → Add Likelihood/Impact/Exposure + residual-after-mitigation columns.
- **A2 (High) Missing risk classes:** clock-domain drift between sample-clock and LTC; **CoreAudio buffer-size renegotiation** mid-session; macOS **App Nap/sleep/Gatekeeper/permission** prompts stealing focus mid-record; **thermal throttle on a fanless-feel mini under sustained 128ch**; **memory fragmentation over multi-day runs**; **JUCE/3rd-party version regressions**; **filesystem choice (APFS vs exFAT)** behavior on power loss; **NTP step / DST** corrupting naming; **solo-dev bus factor** (no second maintainer).

- **A4 (Med) No "accepted risks" register.** Some risks are deliberately accepted for v1 (e.g., no A/B chase). Should be explicit, not silent.
- **A5 (Med) Throughput note understates the real enemy.** Bandwidth is trivial; the note should foreground **latency-spike distribution (p99/p999 write latency)** and **FIFO high-watermark** as the true risk metric.
- **A6 (Low) No risk re-review cadence.** Risks should be revisited at each phase gate.

## Part B — Hardened Section 4

# Section 5 — Data Model: Adversarial Review & Hardening

## Context
Original gave a `Project→Session→Recording(Take)→FileSegment` hierarchy plus Marker/EventLog/Counters, with JSON for config/presets and SQLite for journal/log/markers.

## Part A — Adversarial Review
- **A1 (Critical) No schema versioning / migration** on any entity — early files break on upgrade (echoes §2 A12).
- **A2 (High) Continuous-mode mismatch.** The model is take-centric; reality/live **continuous** recording (per §1.4) has no clean representation — a 12-hour continuous record isn't really a "Take." → Need a `Recording` that supports both bounded takes and open-ended continuous spans, with segments as the real unit.
- **A3 (High) Identity & references fragile.** Implicit IDs; no stable UUIDs, no referential integrity rules, no "what happens to FileSegment if Track is deleted mid-session." → UUIDs everywhere + defined cascade/orphan rules.
- **A4 (High) Time representation ambiguous.** `startTC` + `startSampleSinceMidnight` + `startWallClock` can disagree; no single canonical time anchor or rate/df flags at the right level. → Canonical anchor = sample-count from TimeAuthority; TC/wall derived; store rate+df per Recording.
- **A5 (Med) No append-only/event-sourced separation.** Mixing durable recording facts (immutable once written) with mutable config in the same store risks corrupting history. → Separate **immutable recording ledger** (segments/markers/events, SQLite WAL) from **mutable session config** (JSON).
- **A6 (Med) Channel-map / routing under-modeled** for poly groups and source-name provenance (Dante name vs user name).
- **A7 (Med) No relationship to crash-recovery state machine** — the journal needs explicit per-segment status (open/closing/closed/finalized/orphaned).
- **A8 (Low) Custom metadata fields typing** unspecified (string vs typed); affects iXML mapping + validation.

## Part B — Hardened Section 5
**Two stores:**
- **Mutable config (JSON, versioned):** `Project, Session, DeviceConfig, TrackConfig, MetadataSet+templates, NamingTemplate, FolderTemplate, Preset, Counters.config`.
- **Immutable ledger (SQLite WAL):** `Recording, FileSegment, Marker, EventLogEntry, CounterAllocation`.

**Every entity:** `id: UUID`, `schemaVersion`, `createdAt`. Migrations are forward-only with a version registry.

**Canonical time:** TimeAuthority `sampleCount` is the anchor; `FileSegment.startSample` is authoritative; TC and wall-clock are derived/recorded-for-reference. `Recording { mode: take|continuous, frameRate, dropFrame, startSample, endSample?|open }`.

**Segment lifecycle status:** `open → closing → closed → finalized | orphaned` (drives recovery).

**Referential rules:** TrackConfig edits during record apply to *future* segments only; deleting a track mid-session is blocked while armed; FileSegments reference Track by UUID + a frozen snapshot of name/source at open time (provenance preserved even if track later renamed).

**Source provenance:** TrackConfig stores `{ userName, sourceName (device/Dante), deviceChannel }` distinctly.

**Custom fields:** typed (`string|number|enum|date`) for clean iXML mapping + validation.

## Verification (S5)
Round-trip a session through save→quit→reload with a schema bump and confirm migration; simulate kill-9 mid-segment and confirm ledger status + recovery reconstruct a valid file.

# Section 6 — Audio Pipeline: Adversarial Review & Hardening

## Context
Original: single CoreAudio device (aggregate discouraged), allocate-everything-at-arm, RT callback does delay→gain→bypass→meter/clip tap→FIFO push, with sample-accurate LTC↔sample stamping; meter post-gain.

## Part A — Adversarial Review
- **A1 (Critical) Variable/!=expected buffer size & SR changes** not handled in the callback contract. CoreAudio can deliver variable frame counts and renegotiate. Pre-sized buffers assuming fixed N break. → Callback must handle any frame count ≤ a pre-allocated max; SR change triggers a controlled stop+re-arm, never an in-place realloc.
- **A2 (High) Pre-record buffer absent from the pipeline** (added in §1.5/§3) — must be a continuously-running ring even when not recording. → Standby fills PreRecordBuffer; on record, drain pre-roll into segment with sample-accurate stitch.
- **A3 (High) Delay 0–500ms memory at 128ch.** 500ms @ 192kHz @ 128ch @ float = ~49 MB — fine, but must be allocated at arm and **zeroed** to avoid garbage on first reads; delay-change-while-recording policy undefined (click-free?). → Delay changes apply at segment boundaries or with a crossfade; never realloc in RT.
- **A4 (High) Gain smoothing & dB→linear** must be precomputed and ramped to avoid zipper noise; bypass must be click-free. → One-pole/linear ramp per block; bypass = ramped, not hard switch.
- **A5 (Med) Meter tap point is a policy decision, not a default.** Post-gain reflects recording; many ops want **pre-fader input** to see true input. → Provide both taps; default = recorded (post-gain) with a switch (ties §1 A10).
- **A6 (Med) Multi-device reality.** "Single device" is clean but Dante (64ch) + built-in LTC input is a common 2-device need. Aggregate has drift (R4). → Support a **primary capture device + a separate, independently-clocked LTC/aux input** read on its own callback, correlated via TimeAuthority — avoids aggregating the main capture clock.
- **A7 (Med) Denormals** can spike CPU in delay/gain math. → Enable FTZ/DAZ on the audio thread.
- **A8 (Low) Channel count > device channels / sparse arming** mapping validation belongs here.

## Part B — Hardened Section 6
**Device model:** one **primary clocked capture device**; optional **secondary aux device** (LTC/built-in) on its own callback, never aggregated into the capture clock; both correlated through **TimeAuthority** sample stamping.

**Arm-time allocation (RT-free thereafter):** per-track delay lines (max 500ms @ session SR, zeroed), gain ramp state, capture SPSC FIFOs (seconds of headroom), PreRecordBuffer ring, meter/clip atomics.

**Callback (handles any frame count ≤ max; FTZ/DAZ on):**
```
for each armed track:
  read source channel(s)
  DELAY  (read offset; changes only at boundary/crossfade)
  GAIN   (ramped dB→lin)  [BYPASS = ramped]
  meter+clip tap (post-gain default; pre-fader tap optional)
  push → capture FIFO  (and always → PreRecordBuffer in Standby/Record)
stamp sampleCount → TimeAuthority; publish FIFO HWM + callback duration → Telemetry
```
**SR/buffer change:** detected on message thread → controlled stop + re-arm (no in-place realloc); logged as event.
**No alloc/lock/syscall/exception in callback** (enforced by RT-guard).

## Verification (S6)
Feed variable frame sizes + a mid-session SR change via the mock device; assert no realloc, click-free gain/bypass ramps (spectral check), correct pre-roll stitch, and FTZ active (denormal stress test shows no CPU spike).

# Section 7 — Disk Writing Pipeline: Adversarial Review & Hardening

## Context
Original: small writer pool (2–4), large aligned writes, F_PREALLOCATE, crash-valid header patching, bounded fsync, auto-RF64, gapless rollover (open-before-close), mirror best-effort.

## Part A — Adversarial Review
- **A1 (Critical) Pool→file assignment & ordering** unspecified. 128 files across 2–4 threads needs a defined, contention-free partition and per-file write ordering guarantee (no interleaved partial frames). → Static partition: each file owned by exactly one writer; per-file monotonic write cursor.
- **A2 (High) fsync cadence vs latency spike conflict.** Periodic fsync *is itself* the latency spike (R2). → Decouple: data writes are frequent+buffered; **fsync on a separate cadence/thread**, and header-size patches use small targeted writes; consider `F_FULLFSYNC` only at finalize, not periodically.
- **A3 (High) Crash-valid header strategy under-specified for RF64.** Patching RIFF size + `ds64` mid-stream must be atomic-enough that a crash between data-write and size-patch still yields a recoverable file. → Write data first, advance a *known-good* size marker on a cadence; recovery computes true size from file length if patch lagged.
- **A4 (High) Auto-RF64 decision timing.** Deciding at open from a *projection* can be wrong (rollover changes duration). Switching WAV→RF64 mid-file is impossible. → Default to RF64-capable header when projected OR when rollover policy could exceed 4GB OR sample-rate/channel/duration unknown-but-large; otherwise plain WAV. Bias toward RF64 when uncertain.
- **A5 (Med) Gapless rollover sample-accuracy** needs the exact boundary sample defined by TimeAuthority (esp. for TC-boundary rollover) and metadata/counter carry — but also must not double-count or drop the boundary sample.
- **A6 (Med) Mirror write coupling.** If mirror shares the same FIFO consumption, a slow mirror disk back-pressures primary. → Mirror reads from its **own** tap/FIFO; mirror failure isolates fully.
- **A7 (Med) Disk-full handling** mid-write: must stop the affected destination gracefully, keep the other (mirror) going, finalize a valid truncated file, raise CRITICAL.
- **A8 (Low) Filesystem specifics:** APFS sparse/preallocation behavior, exFAT 4GB limit (forces RF64 / split), volume format detection feeds Destination policy.

## Part B — Hardened Section 7
**Pool:** static file→writer partition (one owner per file, balanced by track count); per-file monotonic cursor; large aligned buffered writes.
**Durability ladder:** (1) frequent buffered data writes; (2) periodic **known-good size marker** + light header patch; (3) `fsync` on a separate low-priority cadence; (4) `F_FULLFSYNC` only at finalize. Recovery trusts file length when a patch lagged.
**Format:** prefer **RF64 header when uncertain/large**; plain WAV only when provably <4GB across the rollover policy; exFAT volumes force RF64/split.
**Rollover (gapless):** TimeAuthority gives the exact boundary sample; open+preallocate next segment before closing current; carry metadata + increment segment counter; boundary sample written exactly once.
**Mirror:** independent tap/FIFO/threads; failure raises status, never back-pressures primary.
**Disk-full:** finalize valid truncated file on the failed volume, continue others, raise CRITICAL + EventLog.

## Verification (S7)
Kill-9 between data-write and size-patch → recovery yields a playable file; fill the primary volume mid-record → mirror continues, primary finalizes valid; force a TC-boundary rollover → byte-exact, no dropped/duplicated boundary sample (golden-file compare).

# Section 8 — LTC Architecture: Adversarial Review & Hardening

## Context
Original: libltc decode off the RT thread via a dedicated tap FIFO; continuous decode; all 7 rates incl. DF/NDF; dropout/discontinuity/rate/midnight detection; sample-accurate `TimeReference`; jam-on-lock; event logging.

## Part A — Adversarial Review
- **A1 (Critical) Sample-accurate audio↔TC correlation is the crux and is hand-waved.** "Stamp the sample index" must define *which* sample maps to *which* LTC frame edge, accounting for the LTC reader's internal latency and the aux-device clock offset (§6 A6). Error here = wrong `TimeReference`. → Define correlation at the **LTC frame-start edge**, store (sampleCount, tc) pairs, and fit a line to reject jitter.
- **A2 (High) Drift between LTC source clock and capture clock** (separate devices) accumulates over hours. A single offset isn't enough. → Continuously estimate **rate ratio** (LTC frames vs capture samples), not just offset; flag drift beyond tolerance.
- **A3 (High) Half-frame / direction / varispeed / jam vs free-run** edge cases. LTC can run backwards (rewind), pause, or varispeed; the decoder must not interpret these as corruption. → State a TC-state model: `locked, free-running(flywheel), paused, reverse, no-signal`; flywheel TC across short dropouts.
- **A4 (High) Midnight vs corruption disambiguation needs precise rule.** `23:59:59:max→00:00:00:00` valid only if it matches expected +1 frame at the detected rate; anything else = discontinuity. → Encode exact expected-next-frame arithmetic per rate/df.
- **A5 (Med) DF/NDF detection & mismatch with project setting.** Must auto-detect drop-frame from the stream (DF flag bit + skipped frame numbers) and reconcile with user setting; mismatch is a WARNING, not silent.
- **A6 (Med) LTC level/quality** (too hot/clipped, too low, wrong channel) — operator needs a confidence/quality indicator, not just lock/no-lock.
- **A7 (Med) What TC anchors a file** when LTC is absent at record start but arrives later, or vice-versa. → Define fallback: time-of-day from system clock with a "TC=ToD (no LTC)" flag; back-fill not allowed (immutable header) but logged.
- **A8 (Low) Frame-rate of 23.976/29.97** are non-integer; sample-per-frame is fractional → must use rational arithmetic, not rounded.

## Part B — Hardened Section 8
**Correlation:** decode on a dedicated thread from an aux/selected-channel tap; for each frame, record `(sampleCount_at_frame_start, tc)` corrected for reader latency; **least-squares fit** over a sliding window yields offset **and rate-ratio** (drift), with jitter rejection. `TimeReference` (samples since midnight) computed from the fit at each segment's start sample using **rational** sample/frame math.
**TC state machine:** `no-signal → locking → locked → flywheel(short loss) → discontinuity/rate-change/paused/reverse`, each emitting EventBus events; flywheel maintains TC across brief dropouts using the last good rate.
**Midnight rule:** rollover accepted only when it equals the exact expected next frame for the detected rate+df; else discontinuity.
**DF/NDF:** auto-detect from stream; mismatch vs project = WARNING.
**Quality:** report LTC level/SNR/lock-confidence to HealthMonitor.
**No-LTC fallback:** anchor to system ToD with explicit flag; immutable, logged.

## Verification (S8)
Synthetic corpus: all 7 rates, drift (±50ppm), short dropouts (flywheel), reverse, pause, varispeed, exact midnight, near-midnight corruption, DF/NDF mismatch, hot/low level. Assert `TimeReference` sample-accuracy vs ground truth and correct state transitions/events.

## Part A — Adversarial Review

- **A8 (Low) Field i18n/encoding** for international crews → UTF-8 in iXML, transliterate for bext.

## Part B — Hardened Section 9
**Model:** typed fields (`string|number|enum|date`), templates, cascade Project→Session→Track→(segment snapshot).
**Embedding (early, not deferred):**

- **iXML (core, early):** PROJECT, SCENE, TAKE, TAPE/ROLL, NOTE, SPEED (rate/df/TimeReference), TRACK_LIST (interleave-correct for poly; single for mono), plus custom fields.
- **Sidecar:** per-daily-folder CSV/JSON sound report.
**Live-edit:** changes apply at **next segment open** via a frozen snapshot; current/written segments immutable.
**Generators:** UMID/OriginatorReference/CodingHistory deterministic from recorder ID + session + timestamp.
**Validation:** configurable required-field policy (warn default, block optional) checked at arm/record.

# Section 10 — Naming Engine: Adversarial Review & Hardening

## Context
Original: one token engine for files + folders; ordered Token|Literal templates; tokens from metadata+counters+TC+device; visual drag-drop builder; live preview; sanitize; duplicate protection; folder templates with auto/ask/disable + midnight handling.

## Part A — Adversarial Review
- **A1 (High) Token resolution timing & immutability.** A filename built from live counters/TC must be resolved **once** at segment open and frozen; re-resolving later (e.g., after a counter change) would rename or mismatch the journal. → Resolve-once-at-open, store the resolved name in the ledger.
- **A2 (High) Duplicate protection across rollover/mirror/crash-recovery.** Auto-suffix must be collision-safe against existing files on disk *and* the in-flight ledger *and* the mirror volume — not just within the session. → Check disk + ledger + mirror; reserve the name atomically before open.
- **A3 (High) Cross-platform / FS sanitization depth.** macOS allows `/` issues, leading dots, length limits (255 bytes UTF-8), reserved names, trailing spaces; exFAT/Windows interop (for delivery) bans `:*?"<>|`. → Sanitize to the **strictest common** profile when delivery target unknown; configurable.
- **A4 (Med) Empty/missing token values** (e.g., no Scene) produce ugly `__` or stray separators. → Define empty-token policy: collapse adjacent separators, optional placeholder, warn.
- **A5 (Med) Poly file naming vs per-track {TrackName}.** A poly file has many tracks; `{TrackName}` is ambiguous. → Token set scoped: file-level tokens vs track-level tokens; `{TrackName}` only valid in mono mode (validate template against mode).
- **A6 (Med) Counter/token interplay with TimeAuthority** for `{TCStart}` — must use the same frozen start sample as the file header (§7/§8).
- **A7 (Low) Template versioning/sharing** across presets; invalid-token detection when a template references a removed custom field.

## Part B — Hardened Section 10
**Resolution:** at segment open, resolve template once against frozen {metadata snapshot, counter allocation, TimeAuthority TCStart, device}; store resolved path in ledger (immutable). `{TCStart}` shares the header's start sample.

- **A5 (Med) Error/event surfacing without alarm fatigue** (§12): the UI needs a non-modal, filterable notification area driven by the EventBus/HealthMonitor, never modal dialogs that block during record.

- **A7 (Med) Live-edit fields while recording** must visibly indicate "applies to next segment" (§9 A3) to avoid operator confusion.
- **A8 (Low) State restoration / window layout persistence** and per-user layouts.

## Part B — Hardened Section 11
**Pattern:** MVVM; ViewModels subscribe to triple-buffered status snapshots + the EventBus; commands out via CommandService only.
**Rendering:** virtualized track list (visible rows only); **GPU-backed single meter view** drawing all visible bars (Metal/OpenGL via JUCE) to hold 128 channels at a configurable 30–60Hz; UI tick is lock-free, alloc-free, latest-wins.

**Notifications:** non-modal, filterable, level-colored area fed by EventBus/HealthMonitor; **no blocking modals during record**.
**Accessibility:** colorblind-safe palettes, high-contrast + large modes, VoiceOver labels, at-a-distance meter mode.
**Windows:** detachable full-screen meter/status view for control-room displays; layout persistence.
**Live-edit affordance:** fields editable during record show an "applies next segment" hint.

# Section 12 — Health Monitoring: Adversarial Review & Hardening

## Context
Original: sampler thread reads CPU/RAM/disk/FIFO/device/LTC/clip; INFO/WARNING/CRITICAL with hysteresis + dwell + rate-limit; clipping classified single/repeated/sustained; silence detection optional default-off; transitions logged.

## Part A — Adversarial Review
- **A1 (High) "Health" must distinguish capture-fatal from cosmetic.** A CRITICAL on the dashboard ≠ a CRITICAL on FIFO overflow. Levels alone don't encode the **failure tier** (§1.6). → Cross every signal with its tier; only Tier-0/1 signals can raise CRITICAL that demands operator action.
- **A2 (High) The real leading indicator is FIFO high-watermark + write p99/p999**, not CPU%. CPU can look fine while a disk stall is about to overflow. → Make FIFO HWM and write-latency percentiles the primary Tier-0 health metrics with predictive (trend) warnings before overflow.
- **A3 (Med) Sampler cadence vs cost.** Sampling SMC temps / disk stats too often is itself overhead; too rarely misses spikes. → Tiered cadence: fast (FIFO/clip, ~10–20Hz via atomics, near-free) vs slow (CPU/RAM/temp/disk, ~1Hz).
- **A4 (Med) Hysteresis params per-signal, not global**; clip vs thermal vs disk need different enter/exit/dwell. → Per-signal threshold config + sane defaults.
- **A5 (Med) Alarm persistence & acknowledgement.** Operators need ack/snooze and a "still active" indication, plus a post-session health summary for the report. → Ack/snooze + latched-until-cleared + session health summary export.
- **A6 (Med) Clip classification needs sample-true detection** (inter-sample peaks / sustained-over-N-samples) from the RT tap, not RMS guesses. → True-peak-ish + consecutive-over-threshold counting in RT; classify off-thread.
- **A7 (Low) Self-health of the monitor** (did the sampler thread stall?) — watchdog.
- **A8 (Low) Thermal action policy:** warn only, or also suggest reducing channels/SR? Document recommended operator action per CRITICAL.

## Part B — Hardened Section 12
**Signal taxonomy:** each signal tagged {tier, cadence-class, level-policy}. Tier-0 leading indicators = **FIFO HWM, write p99/p999, callback-duration p999, xrun count**; Tier-1 = device/LTC status, mirror status, RSS trend; Tier-2 = dashboard/export/network.
**Cadence:** fast loop (atomics, ~10–20Hz) for FIFO/clip/callback; slow loop (~1Hz) for CPU/RAM/temp/disk.
**Leveling:** per-signal enter/exit/dwell hysteresis + rate-limit/coalesce; **predictive** warnings on FIFO/latency trend before overflow.
**Clipping:** RT counts consecutive-over-threshold + peak; off-thread classifies single/repeated/sustained → WARNING-system. Silence detection optional, default-off.
**Operator UX:** ack/snooze, latched-active indication, recommended action per CRITICAL; **session health summary** appended to the sound report.
**Self-watchdog:** monitor liveness check; missed sampler ticks raise a meta-warning.

## Verification (S12)
Inject a slow-disk ramp and confirm a predictive WARNING fires *before* FIFO overflow; verify clip classifier on crafted waveforms; confirm a dashboard crash raises only a Tier-2 INFO/WARNING (never a capture-critical alarm); health summary exports.

# Section 13 — Remote Dashboard: Adversarial Review & Hardening

## Context
Original: read-only embedded HTTP + WebSocket/SSE on a low-priority thread serving the status snapshot; zero RT impact; LAN-bound; optional token auth; assets shareable with in-app web layer.

## Part A — Adversarial Review
- **A1 (High) "Zero RT impact" must be structurally guaranteed, not promised.** An in-process server shares the address space with capture; a bug/leak/crash there can still hurt Tier-0 (contradicts §2 A8). → Run the dashboard **out-of-process (XPC/child)** reading a shared read-only snapshot (shared memory or a one-way pipe); a dashboard crash cannot touch capture.
- **A2 (High) Back-pressure & many clients.** Slow/many WebSocket clients must never queue unbounded in the capture process. → Server holds only the latest snapshot (drop-old), bounded per-client buffers, capped client count.
- **A3 (Med) Security surface.** "Optional token auth" on a broadcast LAN is weak; OB trucks/venues have hostile networks. → Default-on token, bind to explicit interface, optional TLS, read-only by construction (no control path), rate-limit, off-by-default.
- **A4 (Med) Snapshot rate vs cost.** Pushing full 128-ch meters to phones at 60Hz is wasteful/janky on cellular. → Adaptive: status at 1–4Hz, meters summarized (e.g., over/peak flags + reduced rate), client-negotiated.
- **A5 (Med) Discovery & resilience.** mDNS/Bonjour for finding the recorder; auto-reconnect; show "stale data" if the feed drops (never imply all-good when disconnected).
- **A6 (Low) Multi-recorder aggregation** (control room with several recorders) — design the snapshot schema to be recorder-identified for future aggregation.

## Part B — Hardened Section 13
**Process model:** dashboard server is an **isolated child process**; it maps a **read-only latest-snapshot** (shared memory / one-way IPC) published by the engine; no IPC path allows control or write-back. A crash/leak there is invisible to capture.
**Transport:** HTTP for assets, WebSocket/SSE for push; **latest-wins**, bounded per-client buffers, max-client cap; adaptive rates (status 1–4Hz, summarized meters); "stale/disconnected" state shown explicitly.
**Security:** off by default; when on — token auth, explicit interface bind, optional TLS, rate-limit; structurally read-only.
**Discovery:** Bonjour/mDNS advertise; client auto-reconnect.
**Schema:** recorder-identified snapshot (future multi-recorder aggregation).

## Verification (S13)
Kill the dashboard child mid-record → capture unaffected, auto-respawn; connect 50 throttled clients → capture-process memory/CPU flat; pull the network → clients show "stale," not green; confirm no endpoint can issue a command.

# Section 14 — Playback / Spot-Check: Adversarial Review & Hardening

## Context
Original recommendation: integrate as an isolated "Review" mode in the same app (not a separate binary), disabled while recording; own PlaybackEngine + audio path; select take, scrub, solo/group, cached-peak waveform; not a DAW; future companion via journal if needed.

## Part A — Adversarial Review
- **A1 (High) "Same app" vs "Capture is sacred" tension.** Even disabled-during-record, playback code in-process can crash the recorder when *not* recording but a session is loaded. Risk is lower (not armed) but non-zero. → Acceptable for v1 *if* playback strictly cannot initialize while armed/recording and shares no RT structures; revisit out-of-process if instability appears.
- **A2 (High) Device contention** (§3 A11): playback output device vs capture input device. On one interface, opening output may renegotiate the device. → Playback acquires output **only after** capture is fully released; never co-resident; explicit mutual-exclusion in AppCoordinator.
- **A3 (Med) Reading files still being written** (spot-check a prior segment while recording continues) — the original disables all playback during record, but operators often want to verify the *last completed* take while still rolling. → Allow read-only playback of **finalized** segments on a separate output device during record, *if* hardware allows; otherwise disabled. Make it a capability check, not a blanket ban.
- **A4 (Med) Waveform peak generation cost** for long files (multi-hour) — must be cached/lazy/background, never block UI, never compete with writers. → Background peak cache, lazy per-view, throttled, lower priority than writers.
- **A5 (Low) Scope creep guard** — solo/group, scrub is fine; resist adding fades/edits/markers-editing that drift toward a DAW.

## Part B — Hardened Section 14
**Recommendation (confirmed, refined):** **integrated isolated Review mode**, *not* a separate binary — single codebase for a solo dev, shares the session/ledger model directly. **Hard isolation:** separate PlaybackEngine, separate output-device acquisition, no shared RT structures, and AppCoordinator enforces mutual exclusion with capture.
**During record:** allow read-only spot-check of **finalized** segments **only if** a distinct output path is available (capability-gated); never touch in-flight segments or the capture device; default off if hardware can't guarantee isolation.
**Waveform:** background, cached, lazy, throttled, lower-priority-than-writers peak generation.
**Scope:** select/scrub/solo/group + optional waveform. Explicitly not an editor.
**Escalation path:** if in-process playback ever proves destabilizing, promote to an out-of-process companion reading the ledger read-only (reserve the seam).

## Verification (S14)
Confirm playback cannot initialize while armed; on a single interface, verify capture-release→playback-acquire ordering with no glitch; spot-check a finalized segment during an active record on a 2-device rig with zero capture impact; verify peak generation never raises writer latency.
